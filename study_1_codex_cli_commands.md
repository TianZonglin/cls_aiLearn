# Codex CLI 命令教程

本文介绍 Codex CLI 的常用命令，重点说明进入 Codex 交互界面后以 `/` 开头的斜杠命令，例如 `/model`、`/permissions`、`/plan`、`/status`、`/diff`、`/review` 等。

适用对象：

1. 第一次使用 Codex CLI 的同学。
2. 已经安装好 Codex，但不清楚命令如何使用的同学。
3. 想用 Codex 完成代码修改、文档整理、工具开发、课程实践项目的同学。

参考信息：

1. 本机验证版本：`codex-cli 0.137.0`。
2. OpenAI 官方文档：<https://developers.openai.com/codex/cli>
3. Codex CLI 斜杠命令文档：<https://developers.openai.com/codex/cli/slash-commands>
4. Codex CLI 命令行参数文档：<https://developers.openai.com/codex/cli/reference>
5. Codex 模型文档：<https://developers.openai.com/codex/models>

注意：Codex CLI 更新较快，不同版本、不同账号、不同模型提供商能看到的命令和模型可能略有差异。如果实际界面和本文不完全一致，以你本机输入 `/` 后弹出的命令列表为准。

# 1. Codex CLI 的两类命令

使用 Codex 时，需要区分两类命令。

## 1.1 终端命令

终端命令是在 Windows 的 `cmd`、PowerShell、Git Bash 或 macOS/Linux 终端中输入的命令。

例如：

```bat
codex
codex --help
codex --version
codex -m gpt-5.4
codex exec "总结这个项目的结构"
```

这类命令用于：

1. 启动 Codex。
2. 查看帮助和版本。
3. 指定模型、工作目录、图片输入等启动参数。
4. 以非交互方式执行一次任务。

## 1.2 Codex 交互界面中的斜杠命令

进入 Codex 交互界面后，以 `/` 开头输入的命令叫斜杠命令。

例如：

```text
/model
/permissions
/plan
/status
/diff
/review
/compact
/quit
```

这类命令用于：

1. 在不中断会话的情况下切换模型。
2. 调整 Codex 能否读写文件、运行命令、是否需要人工确认。
3. 查看当前会话状态。
4. 检查 Codex 已经改了哪些文件。
5. 压缩上下文，继续长对话。
6. 退出或开启新会话。

一句话区分：

```text
codex、codex --help、codex -m 是终端命令；
/model、/status、/diff 是进入 Codex 之后输入的命令。
```

# 2. 启动 Codex

## 2.1 在项目目录中启动

推荐先进入项目目录，再启动 Codex。

```bat
cd /d E:\Users\Desktop\博士实践课
codex
```

启动后，Codex 会把当前目录作为工作区。它可以读取当前目录中的文件，也可以根据权限设置修改文件或运行命令。

## 2.2 启动时直接带上任务

如果你已经知道要让 Codex 做什么，可以直接在启动命令后写提示词。

```bat
codex "阅读 README.md，帮我提出三个可以改进的地方"
```

适用场景：

1. 任务很短。
2. 不需要多轮讨论。
3. 希望 Codex 启动后直接开始分析。

## 2.3 启动时指定模型

可以用 `-m` 或 `--model` 指定模型。

```bat
codex -m gpt-5.4
```

也可以直接带任务：

```bat
codex -m gpt-5.4 "帮我检查这个项目的目录结构"
```

适用场景：

1. 本次任务需要更强模型。
2. 本次任务只需要快速处理，想换成更快的模型。
3. 临时测试某个模型效果，不想修改配置文件。

注意：可用模型取决于账号、API 配置和模型提供商。官方文档中推荐模型会更新，课程配置中使用的 `gpt-5.4` 只是一个示例。

## 2.4 指定工作目录

如果当前终端不在项目目录，可以用 `-C` 或 `--cd` 指定工作目录。

```bat
codex -C E:\Users\Desktop\博士实践课
```

适用场景：

1. 不想先手动 `cd`。
2. 从脚本中启动 Codex。
3. 同时管理多个项目目录。

## 2.5 附加图片

可以用 `-i` 或 `--image` 把图片附加到初始提示词中。

```bat
codex -i f259d36b7cdee7279e4546caa51c39b0.png "把图片中的教程整理成 Markdown"
```

适用场景：

1. 根据截图写教程。
2. 根据界面截图分析 UI 问题。
3. 根据流程图或报错截图定位问题。

## 2.6 Windows PowerShell 执行策略问题

如果在 PowerShell 中运行 `codex` 出现类似错误：

```text
无法加载文件 codex.ps1，因为在此系统上禁止运行脚本
```

可以改用 `cmd` 运行：

```bat
cmd /c codex
cmd /c codex --help
```

也可以直接打开“命令提示符 cmd”后运行：

```bat
codex
```

# 3. 进入 Codex 后如何使用斜杠命令

## 3.1 打开命令列表

进入 Codex 交互界面后，在输入框中输入：

```text
/
```

Codex 会弹出可用命令列表。继续输入命令名可以过滤列表，例如：

```text
/mo
```

通常会筛选出 `/model`。

## 3.2 执行命令

输入完整命令后按回车。

例如：

```text
/status
```

Codex 会输出当前会话状态。

## 3.3 任务运行中如何排队执行命令

如果 Codex 正在执行任务，你仍然可以输入斜杠命令。官方文档说明：任务运行中可以输入斜杠命令并按 `Tab`，让该命令排队到下一轮执行。

适用场景：

1. Codex 正在长时间运行，你想下一轮查看 `/status`。
2. 本轮完成后立刻执行 `/diff` 看修改。
3. 想在下一轮进入 `/review`。

# 4. `/model`：切换模型

`/model` 用于在交互会话中选择当前使用的模型，有些模型还可以同时选择 reasoning effort，也就是推理强度。

## 4.1 基本用法

在 Codex 输入框中输入：

```text
/model
```

然后按回车。

Codex 会弹出模型选择列表。选择一个模型后，Codex 会在当前会话中使用新模型继续工作。

## 4.2 使用步骤

1. 启动 Codex。
2. 输入 `/model`。
3. 在弹出的列表中选择模型。
4. 如果出现推理强度选项，按任务复杂度选择。
5. 输入 `/status` 验证当前模型是否已经切换。

示例：

```text
/model
/status
```

## 4.3 `/model` 适合什么时候用

### 场景一：任务变复杂了

例如一开始只是让 Codex 整理文档，后来变成让它设计完整工具、修改多处代码、处理测试失败。

这时可以切换到更强模型。

```text
/model
```

选择更强的模型后继续说：

```text
现在请基于前面的讨论，重新梳理实现方案，并开始修改代码。
```

### 场景二：只是做简单任务

如果只是改文档、改 README、整理目录、写简单脚本，可以选择更快或成本更低的模型。

适合任务：

1. 文档格式调整。
2. 简单翻译或润色。
3. 小范围文本替换。
4. 生成命令示例。

### 场景三：对同一个问题比较不同模型效果

你可以先用一个模型让 Codex 生成方案，再用 `/model` 切换模型，让它审查或改进方案。

示例：

```text
/model
```

然后输入：

```text
请审查上一个方案，指出可能不稳妥的地方，并给出更保守的实现步骤。
```

## 4.4 `/model` 和 `codex -m` 的区别

`codex -m` 是启动前指定模型。

```bat
codex -m gpt-5.4
```

`/model` 是进入 Codex 后，在当前会话中切换模型。

```text
/model
```

对比：

| 方式 | 使用位置 | 是否需要重新启动 Codex | 适合场景 |
|---|---|---|---|
| `codex -m 模型名` | 终端 | 需要启动新会话 | 一开始就知道要用哪个模型 |
| `/model` | Codex 交互界面 | 不需要 | 会话中途根据任务复杂度调整 |

## 4.5 推理强度如何选择

如果模型支持 reasoning effort，通常可以按下面原则选择：

| 推理强度 | 适合任务 | 特点 |
|---|---|---|
| low | 简单问答、改错别字、小范围文档修改 | 更快 |
| medium | 一般代码修改、常规分析 | 平衡速度和质量 |
| high | 多文件修改、复杂调试、架构设计、重要审查 | 更细致，但更慢 |

课程实践中建议：

1. 写 README、整理教程：`medium` 通常足够。
2. 修改代码、排查 bug：建议 `medium` 或 `high`。
3. 设计完整 AI 工具：建议先 `high` 做方案，再用 `medium` 执行简单修改。

# 5. `/permissions`：调整权限和审批模式

`/permissions` 用于设置 Codex 可以自动做什么、哪些操作需要你确认。

## 5.1 基本用法

```text
/permissions
```

按回车后，Codex 会显示权限选项，例如更自动的模式，或更保守的只读模式。具体名称会随版本和配置变化。

## 5.2 常见权限思路

| 模式思路 | 含义 | 适合场景 |
|---|---|---|
| Read Only / 只读 | Codex 只读文件，不直接修改 | 第一次看项目、让它解释代码、做方案 |
| Auto / 自动 | Codex 可以更主动地修改文件和运行命令 | 小项目、练习项目、你信任当前操作 |
| 需要确认 | 某些命令或写操作前需要你批准 | 正式项目、重要文件、敏感操作 |

## 5.3 适用场景

### 场景一：先让 Codex 读项目，不要动文件

先输入：

```text
/permissions
```

选择只读或较保守的模式，然后提问：

```text
请阅读这个项目，说明目录结构和主要文件作用，不要修改任何文件。
```

### 场景二：确认方案后允许 Codex 修改

当你已经认可修改方案，可以再次输入：

```text
/permissions
```

切换到允许写入的模式，然后说：

```text
按刚才的方案修改文件，修改后说明改了哪些地方。
```

### 场景三：运行命令前需要人工确认

对于可能影响环境的命令，例如安装依赖、删除文件、重置 Git，建议保持需要确认的模式。

高风险命令示例：

```bat
npm install
git reset --hard
del /s
```

建议：课程练习中可以适当使用自动模式，但正式项目中应优先使用保守权限。

# 6. `/plan`：先规划再执行

`/plan` 用于让 Codex 先进入规划模式，先给方案，不急着改文件。

## 6.1 基本用法

```text
/plan
```

也可以直接带上问题：

```text
/plan 帮我设计一个课程资料自动整理工具，要求四周内能完成
```

## 6.2 适用场景

`/plan` 适合用在任务还不清楚、风险较高或需要先沟通的时候。

常见场景：

1. 新建一个工具项目。
2. 多文件代码重构。
3. 需要先选技术路线。
4. 不确定是否会影响已有功能。
5. 希望老师或同学先看方案再执行。

## 6.3 示例

```text
/plan 我想做一个课程资料自动整理工具，请给出功能范围、技术方案、目录结构和第一步实现任务。
```

Codex 可能会输出：

1. 功能边界。
2. 输入输出格式。
3. 技术选型。
4. 文件结构。
5. 分阶段任务。
6. 风险和注意事项。

你确认后再说：

```text
按照这个方案，先创建最小可运行版本。
```

# 7. `/status`：查看当前会话状态

`/status` 用于查看 Codex 当前会话的关键状态。

## 7.1 基本用法

```text
/status
```

## 7.2 通常可以看到什么

不同版本显示内容略有差异，通常包括：

1. 当前模型。
2. 当前权限或审批模式。
3. 当前工作目录。
4. 可读写目录。
5. token 或上下文使用情况。
6. 远程连接状态。

## 7.3 适用场景

### 场景一：确认模型是否切换成功

```text
/model
/status
```

### 场景二：确认 Codex 是否在正确目录工作

如果你担心 Codex 改错目录，输入：

```text
/status
```

查看当前工作根目录。

### 场景三：检查上下文是否快满

长时间对话后，输入：

```text
/status
```

如果上下文占用较高，可以使用 `/compact`。

# 8. `/diff` 与 `/review`：检查修改和代码审查

这两个命令经常配合使用。

## 8.1 `/diff`：查看 Codex 改了什么

基本用法：

```text
/diff
```

作用：

1. 查看 Git diff。
2. 查看已修改但未提交的文件。
3. 查看未跟踪的新文件。
4. 在提交前确认修改是否符合预期。

适用场景：

1. Codex 修改完文件后。
2. 准备 `git add` 和 `git commit` 前。
3. 你怀疑 Codex 改了不该改的内容。

示例流程：

```text
请把 README 的课程安排写得更清楚。
/diff
```

## 8.2 `/review`：让 Codex 审查当前修改

基本用法：

```text
/review
```

作用：

1. 审查当前工作区修改。
2. 找潜在 bug。
3. 找遗漏测试。
4. 提醒可能的行为变化。

适用场景：

1. 代码修改完成后。
2. 提交前想多一次检查。
3. 多文件修改后想确认有没有遗漏。

示例流程：

```text
/diff
/review
```

如果 `/review` 提出问题，可以继续说：

```text
请修复 review 中提到的第 1 个问题。
```

## 8.3 推荐提交前流程

```text
/status
/diff
/review
```

然后在终端中提交：

```bat
git status
git add .
git commit -m "描述本次修改"
git push
```

# 9. `/compact`、`/clear`、`/new`：管理上下文

长时间使用 Codex 后，聊天记录会越来越长。上下文过长会影响速度，也可能让 Codex更难抓住当前重点。

## 9.1 `/compact`：压缩上下文

基本用法：

```text
/compact
```

作用：把前面的长对话总结成较短摘要，释放上下文空间，同时保留关键任务信息。

适用场景：

1. 已经聊了很久。
2. Codex 开始变慢。
3. 当前任务还没做完，但前面细节太多。
4. `/status` 显示上下文占用较高。

建议用法：

```text
/compact
```

然后继续：

```text
基于刚才压缩后的上下文，继续完成剩余任务。
```

## 9.2 `/clear`：清空终端显示并开始新聊天

基本用法：

```text
/clear
```

作用：清理当前可见界面，并开启更干净的聊天状态。

适用场景：

1. 你想从头问一个问题。
2. 当前对话已经不重要。
3. 终端显示太乱。

## 9.3 `/new`：在同一个 CLI 中开启新对话

基本用法：

```text
/new
```

作用：不退出 Codex，但开始一个新的对话上下文。

适用场景：

1. 一个任务已经完成，准备做另一个任务。
2. 不想让前一个任务影响新任务。
3. 仍然在同一个项目目录中工作。

## 9.4 什么时候用哪个

| 命令 | 是否保留摘要 | 是否适合继续当前任务 | 适合场景 |
|---|---|---|---|
| `/compact` | 保留 | 适合 | 长任务继续做 |
| `/clear` | 通常不强调保留 | 适合轻量重置 | 终端太乱、想清屏 |
| `/new` | 开新上下文 | 不适合继续旧任务 | 换一个新任务 |

# 10. `/mention`、`/mcp`、`/plugins`、`/skills`：补充上下文和工具

## 10.1 `/mention`：把文件或目录指给 Codex

基本用法：

```text
/mention
```

作用：把某个文件或目录附加到当前对话，让 Codex 优先阅读或引用。

适用场景：

1. 让 Codex 阅读某个配置文件。
2. 让 Codex 修改指定文档。
3. 避免 Codex 在大量文件中找错目标。

示例：

```text
/mention
```

选择 `softwares_install.md` 后输入：

```text
请只根据这个文件内容，帮我检查目录链接是否合理。
```

## 10.2 `/mcp`：查看 MCP 工具

基本用法：

```text
/mcp
```

查看详细信息：

```text
/mcp verbose
```

作用：查看当前 Codex 可调用的 MCP 工具，例如文档检索、浏览器、数据库、第三方服务等。

适用场景：

1. 检查某个 MCP 是否配置成功。
2. 确认 Codex 是否能访问外部工具。
3. 调试工具调用失败。

## 10.3 `/plugins`：查看和管理插件

基本用法：

```text
/plugins
```

作用：查看已安装或可发现的插件，检查插件能力，启用或禁用插件。

适用场景：

1. 需要浏览器能力。
2. 需要文档、表格、PPT 等插件。
3. 想确认某个插件是否可用。

## 10.4 `/skills`：选择技能

基本用法：

```text
/skills
```

作用：选择某类任务的专门技能，让 Codex 按更合适的流程工作。

适用场景：

1. 写前端页面。
2. 处理 Word、Excel、PPT。
3. 分析图片。
4. 生成媒体素材。

# 11. `/resume`、`/fork`、`/side`：继续、分叉和旁路讨论

## 11.1 `/resume`：恢复历史会话

基本用法：

```text
/resume
```

作用：从保存的会话列表中恢复之前的对话。

适用场景：

1. 之前任务没做完。
2. 想继续上一次 Codex 工作。
3. 不想重新解释项目背景。

也可以在终端中使用：

```bat
codex resume
```

## 11.2 `/fork`：分叉当前会话

基本用法：

```text
/fork
```

作用：从当前对话复制出一个新分支，用来尝试另一个思路。

适用场景：

1. 当前方案可行，但想比较另一个方案。
2. 不想破坏当前对话上下文。
3. 想让 Codex 试验不同实现路径。

## 11.3 `/side` 或 `/btw`：开启旁路讨论

基本用法：

```text
/side
```

也可以带问题：

```text
/side 这个实现方案有没有明显风险？
```

作用：临时开一个旁路讨论，不干扰主任务上下文。

适用场景：

1. 主任务正在进行，但你突然想问一个小问题。
2. 想检查一个想法，但不希望影响主线。
3. 想让 Codex 临时评估风险。

# 12. 其他常用斜杠命令速查

| 命令 | 作用 | 适用场景 |
|---|---|---|
| `/fast` | 开启、关闭或查看 Fast 服务层级 | 当前模型支持 Fast，且希望更快响应 |
| `/personality` | 调整 Codex 回复风格 | 想让回答更简洁、更协作或关闭风格 |
| `/copy` | 复制 Codex 最近一次完整回复 | 需要把方案或代码复制到文档中 |
| `/approve` | 对最近一次被自动审查拒绝的操作批准重试 | 你确认该操作可以执行 |
| `/ps` | 查看后台终端任务 | 有长时间运行命令时检查进度 |
| `/stop` | 停止后台终端任务 | 命令卡住或不再需要继续运行 |
| `/raw` | 切换原始滚动回显模式 | 需要更方便地复制终端输出 |
| `/init` | 生成 `AGENTS.md` 脚手架 | 想为项目写长期规则和协作说明 |
| `/logout` | 登出 Codex | 公共电脑、账号切换 |
| `/quit` | 退出 Codex | 完成任务后退出 |
| `/exit` | 退出 Codex，等同于 `/quit` | 习惯输入 exit 的用户 |
| `/keymap` | 配置 TUI 快捷键 | 想自定义键盘操作 |
| `/vim` | 开关 Vim 编辑模式 | 习惯 Vim 输入方式 |
| `/statusline` | 配置底部状态栏 | 想显示模型、上下文、Git 分支等信息 |

| `/debug-config` | 查看配置层和策略诊断 | 配置文件生效异常、权限策略异常 |
| `/feedback` | 发送反馈和诊断日志 | 遇到 Codex CLI 问题需要反馈 |

注意：部分命令只在特定平台、模型、配置或实验功能开启时出现。例如 `/fast` 取决于当前模型是否提供 Fast 层级，`/sandbox-add-read-dir` 是 Windows 相关命令。

# 14. 常见使用场景

## 14.1 场景一：让 Codex 先读项目，不要修改

```text
/permissions
```

选择只读或保守模式，然后输入：

```text
请阅读当前项目，说明文件结构、主要内容和后续可以完善的地方。不要修改文件。
```

适合第一次打开项目时使用。

## 14.2 场景二：写一个新文档

```text
/model
```

选择合适模型后输入：

```text
请新建 study_2_project_ideas.md，写一份 AI 工具选题说明，面向博士实践课同学。
```

完成后检查：

```text
/diff
```

## 14.3 场景三：先做方案，再让 Codex 修改

```text
/plan 我想做一个课程资料自动整理工具，请先给出功能范围、文件结构和开发步骤。
```

确认方案后：

```text
按这个方案创建最小可运行版本。
```

## 14.4 场景四：长任务中途压缩上下文

```text
/status
/compact
```

然后继续：

```text
继续完成剩余功能，并在最后说明修改了哪些文件。
```

## 14.5 场景五：提交前检查

在 Codex 中：

```text
/diff
/review
```

在终端中：

```bat
git status

git push
```

## 14.6 场景六：切换模型做二次审查

```text
/model
```

选择更强模型后：

```text
请审查当前实现，重点看有没有遗漏的错误处理、边界情况和文档不一致。
```

# 15. 常见问题

## 15.1 输入 `/model` 后看不到想要的模型

可能原因：

1. 当前账号没有该模型权限。
2. API Key 对应的服务商不支持该模型。
3. 当前 Codex 版本较旧。
4. 配置文件限制了可用模型。

可以尝试：

```text
/status
```

或在终端运行：

```bat
codex --version
codex update
```

## 15.2 Codex 改错目录了怎么办

先用：

```text
/status
```

确认工作目录。

下次启动时建议先进入目录：

```bat
cd /d E:\Users\Desktop\博士实践课
codex
```

或者使用：

```bat
codex -C E:\Users\Desktop\博士实践课
```

## 15.3 Codex 改了文件，但我不知道改了什么

输入：

```text
/diff
```

如果需要审查：

```text
/review
```

## 15.4 对话太长，Codex 开始混乱

输入：

```text
/compact
```

如果要换新任务，输入：

```text
/new
```

## 15.5 PowerShell 里运行 Codex 报脚本策略错误

可以使用 `cmd`：

```bat
cmd /c codex
```

或者打开“命令提示符 cmd”后直接输入：

```bat
codex
```

## 15.6 不知道某个命令是否可用

进入 Codex 后输入：

```text
/
```

查看弹出的命令列表。也可以输入命令前几个字母筛选，例如：

```text
/sta
```

通常可以筛选出 `/status`。

# 推荐学习顺序

初学者不需要一次记住所有命令，建议先掌握以下 8 个：

1. `/model`：切换模型。
2. `/permissions`：调整能否自动修改和运行命令。
3. `/plan`：先让 Codex 出方案。
4. `/status`：确认模型、目录、权限和上下文。
5. `/diff`：查看文件改动。
6. `/review`：审查当前改动。
7. `/compact`：长对话压缩上下文。
8. `/quit`：退出 Codex。

掌握这几个命令后，就可以比较稳妥地用 Codex 完成课程实践项目。
