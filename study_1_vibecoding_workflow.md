# Vibe Coding 工具开发工作流

本文用于说明：使用 Codex 辅助开发一个工具时，应该如何从想法、调研、设计、拆分任务、执行开发、调试微调，一直推进到提交 GitHub 和整理项目文档。

核心原则是：先想清楚，再让 AI 写；先写设计文档，再拆步骤；每一步都要有明确输入、输出和校验；调试阶段持续保留上下文；阶段性完成后及时提交。

# 一、阶段 1：构想与调研

## 1-1. 构想、具体（大脑想）

开发工具的第一步不是打开 Codex，也不是马上写代码，而是先把“我到底想做什么”想清楚。

很多项目失败，不是因为代码写不出来，而是因为一开始目标太虚。例如：

- “我想做一个 AI 学习工具”太宽泛。
- “我想做一个可以上传课程 PDF，自动提取知识点、生成复习题和 Markdown 笔记的工具”就具体得多。

构想阶段至少要回答以下问题：

| 问题 | 说明 | 示例 |
|---|---|---|
| 用户是谁 | 谁会使用这个工具 | 自己、同学、老师、实验室成员 |
| 解决什么问题 | 不要只说功能，要说痛点 | 手动整理文献太慢、会议纪要容易遗漏 |
| 输入是什么 | 用户给工具什么材料 | PDF、Excel、文本、图片、表单内容 |
| 处理过程是什么 | 工具需要如何加工输入 | 提取、分类、总结、计算、生成图表 |
| 输出是什么 | 用户最终得到什么 | Markdown、Excel、网页结果、报告、图表 |
| 使用频率如何 | 是一次性工具还是反复使用 | 每周整理文献、每天记录实验 |
| 最小可用版本是什么 | 第一个版本最少做到什么 | 能上传一个文件并生成一份结果 |

建议在纸上或普通 Markdown 中先写出一句话版本：

```text
我想开发一个【工具名称】，它可以让【目标用户】输入【输入内容】，自动完成【核心处理】，最后输出【结果】，从而解决【具体问题】。
```

示例：

```text
我想开发一个课程资料整理工具，它可以让学生上传课程 PDF 和课堂笔记，自动提取主题、关键词、摘要和待复习问题，最后输出 Markdown 学习笔记，从而减少手工整理资料的时间。
```

这一阶段不要追求完美，重点是把目标从模糊想法压缩成一个可开发的小工具。

## 1-2. 调研、相似产品/功能（动手查阅）

有了初步构想后，需要动手查阅相似产品或相似功能。调研不是为了复制，而是为了明确：

- 别人已经做了什么。
- 哪些功能是用户自然期待的。
- 哪些功能对第一版来说太复杂。
- 自己的工具应该保留什么、舍弃什么。

可以调研以下来源：

| 来源 | 适合看什么 |
|---|---|
| GitHub | 开源项目结构、README、功能范围、技术栈 |
| 产品官网 | 产品定位、核心卖点、界面流程 |
| B 站 / YouTube | 实际使用演示 |
| 小红书 / 知乎 / 博客 | 用户痛点、使用场景、替代方案 |
| 论文或课程案例 | 专业场景中的真实需求 |

调研时建议记录成表格：

| 产品/项目 | 主要功能 | 优点 | 不足 | 可借鉴点 |
|---|---|---|---|---|
| 产品 A | 上传 PDF 并总结 | 流程简单 | 不支持导出 | 可借鉴上传和摘要流程 |
| 项目 B | 生成图表报告 | 输出专业 | 配置复杂 | 可借鉴报告结构 |

调研完成后，要形成一个结论：

```text
我的工具第一版不做大而全系统，只保留最关键的输入、处理、输出流程。
```

例如，文献阅读工具第一版可以只做：

1. 上传 PDF。
2. 提取文本。
3. 生成摘要、研究问题、方法、结论。
4. 导出 Markdown。

暂时不做：

1. 多用户登录。
2. 云端同步。
3. 复杂知识图谱。
4. 支付功能。
5. 高级权限管理。

# 二、阶段 2：规划与设计文档

阶段 2 建议使用 `/plan` 模式。这个阶段的目标不是让 Codex 直接写代码，而是让 Codex 帮助你把想法变成清晰、完整、可执行的设计方案。

## 2-1. 产品形态、技术栈（通过 /plan 模式辅助理清）

产品形态决定用户如何使用这个工具。常见形态包括：

| 产品形态 | 适合场景 | 优点 | 缺点 |
|---|---|---|---|
| 命令行工具 | 文件批处理、自动化脚本 | 简单、开发快 | 不适合普通用户 |
| Web 页面工具 | 上传、预览、交互式操作 | 直观、适合展示 | 前后端复杂度更高 |
| 桌面工具 | 本地文件处理、离线使用 | 接近普通软件 | 打包和兼容性复杂 |
| 浏览器插件 | 网页内容提取和增强 | 使用场景明确 | 权限和浏览器适配复杂 |
| Python 脚本 | 数据分析、实验流程 | 容易实现 | 用户体验较弱 |

对课程实践来说，推荐优先选择：

- Web 页面工具：适合展示和演示。
- Python 脚本工具：适合数据处理和自动化。
- Node.js 小工具：适合 Markdown、网页、API 处理。

技术栈选择要遵循“够用即可”的原则。不要为了炫技选择自己完全不熟悉、项目又不需要的复杂技术。

示例提示词：

```text
/plan 我想开发一个课程资料整理工具，用户上传 PDF 或 TXT，工具自动生成摘要、关键词、复习题，并导出 Markdown。请帮我判断适合做成命令行工具、Web 工具还是桌面工具，并推荐十天内可完成的技术栈。
```

规划时需要让 Codex 输出：

1. 推荐产品形态。
2. 推荐技术栈。
3. 不推荐方案及原因。
4. 十天内可完成的最小版本。
5. 后续可扩展方向。

## 2-2. 核心功能、辅助功能完善（通过 /plan 模式辅助理清）

确定产品形态后，要区分核心功能和辅助功能。

核心功能是没有它工具就不成立的功能。辅助功能是让工具更好用、更完整，但第一版可以延后的功能。

例如，一个“课程资料整理工具”可以这样拆：

| 类型 | 功能 | 是否第一版必须 |
|---|---|---|
| 核心功能 | 上传课程文件 | 必须 |
| 核心功能 | 读取文本内容 | 必须 |
| 核心功能 | 生成摘要和关键词 | 必须 |
| 核心功能 | 生成 Markdown 笔记 | 必须 |
| 辅助功能 | 历史记录 | 可选 |
| 辅助功能 | 多文件批处理 | 可选 |
| 辅助功能 | 登录账号 | 暂不做 |
| 辅助功能 | 云端同步 | 暂不做 |

建议使用 MoSCoW 方法整理优先级：

| 优先级 | 含义 |
|---|---|
| Must have | 第一版必须有 |
| Should have | 应该有，但可以简化 |
| Could have | 有更好，没有也能用 |
| Won't have now | 当前阶段明确不做 |

示例提示词：

```text
/plan 请基于上面的工具想法，帮我拆分 Must have、Should have、Could have、Won't have now 四类功能。要求第一版十天内能完成，不要加入登录、支付、复杂权限等大型系统功能。
```

## 2-3. 具体完整的详细设计文档（通过 /plan 模式辅助理清）

设计文档是后续开发的根。没有设计文档，后面每一轮问答都容易漂移。

一个合格的 `design.md` 至少应包含：

| 模块 | 内容 |
|---|---|
| 项目背景 | 为什么做这个工具 |
| 目标用户 | 谁使用 |
| 核心场景 | 用户在什么情况下使用 |
| 产品形态 | Web、命令行、脚本、插件等 |
| 技术栈 | 前端、后端、数据库、文件处理、AI API |
| 功能清单 | 核心功能和辅助功能 |
| 页面或命令流程 | 用户如何一步步操作 |
| 数据结构 | 输入、处理中间数据、输出格式 |
| 文件结构 | 项目目录和主要文件 |
| API 设计 | 如果有后端，需要列出接口 |
| 错误处理 | 上传失败、解析失败、AI 调用失败怎么办 |
| 验收标准 | 做到什么程度算完成 |
| 暂不实现内容 | 明确边界，防止范围膨胀 |

示例提示词：

```text
/plan 请帮我把这个工具整理成完整 design.md 的详细设计文档草案。要求包含项目背景、目标用户、使用流程、功能清单、技术栈、目录结构、数据结构、接口设计、错误处理、验收标准和暂不实现内容。先不要写代码。
```

## 2-4. 问答内容落盘，形成文档 design.md

当 `/plan` 阶段讨论清楚后，需要把问答结果写入 `design.md`。

这一步非常重要，因为 Codex 的上下文会变长、会压缩，也可能之后换新会话。如果没有文档落盘，项目知识只存在对话里，很容易丢失。

建议对 Codex 说：

```text
请把刚才确认的方案整理成 design.md，写入项目根目录。要求结构清晰，后续可以直接根据这个文档拆分开发步骤。
```

`design.md` 写好后，不要急着开发，先自己读一遍，重点检查：

1. 目标是否具体。
2. 第一版范围是否过大。
3. 输入和输出是否明确。
4. 每个功能是否能验收。
5. 暂不实现内容是否写清楚。

# 三、阶段 3：拆分开发步骤

阶段 3 开始进入 default 模式。此时已经有 `design.md`，目标是把一个大项目拆成若干个可执行的 `step-X.md`。

## 3-1. 将 design.md 拆分为若干步输入输出明确的提示词，写入 step-X.md

每个 `step-X.md` 不是普通任务说明，而应该是可以直接发给 Codex 执行的一轮开发提示词。

一个好的步骤文件应该包含：

| 部分 | 说明 |
|---|---|
| 本步目标 | 这一轮只做什么 |
| 输入 | 本步依赖哪些已有文件、配置、接口 |
| 输出 | 本步完成后应该生成或修改哪些文件 |
| 不做什么 | 明确禁止扩大范围 |
| 实现要求 | 技术细节、边界情况、风格要求 |
| 校验方式 | 运行什么命令、检查什么页面、看什么结果 |
| 完成标准 | 怎样判断本步完成 |

示例 `step-1.md` 结构：

```markdown
# step-1：项目初始化

## 目标

根据 design.md 初始化项目结构，安装必要依赖，创建最小可运行版本。

## 输入

- design.md
- 当前空项目目录

## 输出

- package.json
- src/ 目录
- README 初稿
- 可启动的开发服务器

## 不做

- 不实现完整业务功能
- 不接入真实 AI API
- 不做登录注册

## 校验

- npm install 成功
- npm run dev 成功
- 浏览器可以打开首页
```

推荐提示词：

```text
请读取 design.md，把整个开发过程拆分成 6-10 个步骤，每个步骤写入 step-1.md、step-2.md 等文件。每个步骤必须包含目标、输入、输出、不做内容、实现要求、校验方式和完成标准。
```

拆分时要注意：

1. 每一步都应该可以独立完成和验证。
2. 不要把“完成整个前端”塞进一个步骤。
3. 不要把“修复所有问题”作为步骤目标。
4. 越靠前的步骤越应该基础，例如项目初始化、数据结构、核心流程。
5. 越靠后的步骤越适合做界面优化、导出、错误处理、文档。

## 3-2. 检查每个步骤 md 文件，重点看输入输出的校验是否完善正确

拆完步骤后，需要检查每个 `step-X.md` 是否真的可执行。

检查重点：

| 检查项 | 合格标准 |
|---|---|
| 目标是否单一 | 一轮只解决一类问题 |
| 输入是否明确 | Codex 知道要读哪些文件 |
| 输出是否明确 | Codex 知道要改哪些文件 |
| 校验是否具体 | 不是“看起来正常”，而是有命令或页面行为 |
| 边界是否清楚 | 明确不做哪些功能 |
| 顺序是否合理 | 后一步不依赖尚未完成的内容 |

可以让 Codex 自查：

```text
请检查 step-1.md 到 step-8.md，重点审查每一步的输入、输出、校验方式和完成标准是否明确。如果发现步骤过大、依赖不清或验收不清，请直接修改这些 step 文件。
```

自己检查时，可以重点问三个问题：

1. 如果把这个文件单独发给 Codex，它能不能知道要做什么？
2. 做完后，我能不能判断它是否完成？
3. 如果这一步失败，我能不能定位失败在哪里？

# 四、阶段 4：逐步执行与初步跑通

## 4-1. 分步骤逐个执行 step-X.md，每轮完成让 Codex 自动检验

进入执行阶段后，不要一次性把所有 `step-X.md` 都丢给 Codex。推荐一轮只执行一个步骤。

例如：

```text
请执行 step-1.md。完成后请按照其中的校验方式自动检查，并说明修改了哪些文件、校验结果如何、还有哪些风险。
```

每一轮结束时，让 Codex 输出：

1. 本轮完成了什么。
2. 修改了哪些文件。
3. 执行了哪些校验。
4. 校验是否通过。
5. 如果未通过，原因是什么。
6. 下一步建议执行哪个 step。

执行时建议遵守：

- 不要跳步。
- 不要一轮做多个大步骤。
- 每轮都让 Codex 自动跑测试或启动检查。
- 每轮完成后用 `/diff` 看文件变化。
- 关键步骤完成后可以用 `/review` 让 Codex 做一次代码审查。

## 4-2. 全部执行完毕，功能大致正常即可，随即进入调试阶段

当所有 `step-X.md` 执行完后，不要期待工具已经非常成熟。这个阶段的目标是“主流程大致可用”。

大致正常通常指：

1. 项目能启动。
2. 主要页面或命令能打开。
3. 核心输入可以提交。
4. 能得到基本输出。
5. 没有明显崩溃。

不要求：

1. 所有界面都美观。
2. 所有异常都处理完美。
3. 所有边缘情况都覆盖。
4. 所有文档都完整。

此时应立即进入调试阶段。调试阶段比初始开发更长，也更接近真实开发。

## 4-3. 每次进入 Codex 后 /resume 原有问答 session

从调试阶段开始，上下文非常重要。Codex 需要知道：

- 之前为什么这样设计。
- 哪些方案已经试过。
- 哪些 bug 已经修过。
- 哪些功能暂时不做。
- 当前项目处于什么阶段。

因此每次重新进入 Codex 后，优先使用 `/resume` 回到原有问答 session。

推荐流程：

```text
1. 打开项目目录。
2. 启动 codex。
3. 使用 /resume 找到原来的项目会话。
4. 继续描述当前要调试的问题。
```

如果不小心开了新会话，应先让 Codex 读取关键文档：

```text
请先读取 design.md、README.md、step-*.md 和当前项目结构，理解项目背景后再继续。不要立即修改文件。
```

# 五、阶段 5：调试、微调与版本管理

## 5-1. Web 项目打开页面，每轮包括 3-5 处微调内容

如果是 Web 项目，调试不能只靠想象，一定要打开页面实际看。

常见调试方式：

1. 启动开发服务器。
2. 打开浏览器页面。
3. 试一遍核心流程。
4. 记录界面问题、交互问题、报错信息。
5. 一轮整理 3-5 个微调点发给 Codex。

微调时不要每次只说一处，例如：

```text
请微调当前 Web 页面：
1. 顶部标题太大，降低字号并减少上下留白。
2. 上传区域不够明显，增加边框和拖拽提示。
3. 结果区域内容太拥挤，增加段落间距。
4. 生成按钮在移动端会换行，请优化按钮宽度。
5. 错误提示颜色太刺眼，请调整为更克制的红色。
完成后请启动检查并说明修改文件。
```

这样效率更高，也能减少上下文浪费。

但每轮也不要塞太多问题。超过 5 个微调点，Codex 容易漏改或引入新问题。

## 5-2. 复杂调整每次独占一轮提问

复杂问题要单独处理，不要和普通微调混在一起。

复杂调整包括：

- 文件上传逻辑错误。
- AI API 调用失败。
- 数据结构需要重构。
- 页面状态管理混乱。
- 导出文件格式不正确。
- 性能明显有问题。
- 跨平台路径兼容问题。

复杂调整提示词示例：

```text
当前文件上传后偶尔解析失败。请专门排查这个问题，不要改 UI。请先阅读相关上传、解析和错误处理代码，说明可能原因，再实施修复。修复后请补充或运行校验。
```

复杂问题的建议流程：

1. 让 Codex 先读代码并说明原因。
2. 再让 Codex 修改。
3. 修改后让 Codex 自测。
4. 如果还失败，继续同一个问题追问。
5. 问题解决后再进入下一类问题。

## 5-3. 达到某一完成状态时及时提交 GitHub，避免频繁提交

GitHub 提交是保存阶段成果，不是每改一行就提交。

适合提交的时机：

| 时机 | 示例 |
|---|---|
| 初始化完成 | 项目能启动 |
| 核心功能跑通 | 上传、处理、输出完整 |
| 一轮调试完成 | 主要 bug 修复 |
| 文档完成 | README、design.md 更新 |
| 演示版本完成 | 可以给别人试用 |

不建议提交的时机：

- 刚改了一半还不能运行。
- 明知道有严重报错。
- 只调整了一个按钮颜色。
- 临时调试代码还没清理。

提交前建议做三件事：

1. 运行测试或启动检查。
2. 使用 `/diff` 查看改动。
3. 使用 `/review` 做一次提交前审查。

提交信息建议清楚表达阶段成果：

```text
feat: initialize course material organizer
feat: implement markdown export workflow
fix: handle pdf parsing failure
docs: add design and usage documentation
```

## 5-4. 完整 Vibe 一个成熟工具一般会经历 100-300 轮问答，要控制预期

一个成熟工具不是十几轮问答就能完成的。尤其是 Web 工具、包含文件上传、AI 调用、导出、状态管理的项目，真实迭代轮次会比较多。

一般可以这样理解：

| 阶段 | 可能轮次 |
|---|---|
| 构想与规划 | 5-20 轮 |
| 设计文档 | 5-15 轮 |
| 步骤拆分 | 3-10 轮 |
| 初始开发 | 20-60 轮 |
| 调试修复 | 30-100 轮 |
| UI 微调 | 20-80 轮 |
| 文档与提交 | 5-15 轮 |

所以完整 Vibe 一个成熟工具经历 100-300 轮问答是正常的。

要控制预期：

- 第一版只追求能用。
- 第二版再追求好用。
- 第三版再追求稳定和美观。
- 不要一开始就要求“像商业产品一样完整”。

## 5-5. 轮次过大可以新建 session，但应在一轮提交之后

长对话会带来上下文压力。轮次过大时，可以新建一个 session，但不要在代码改到一半时换会话。

推荐换 session 的时机：

1. 当前功能已经完成。
2. 已经运行过校验。
3. 已经提交 GitHub。
4. README 或 design.md 已经更新。
5. 下一步是一个相对独立的新功能或新阶段。

新 session 开始时，应让 Codex 先读项目资料：

```text
请先读取 README.md、design.md、QA_History.md、当前目录结构和最近 Git 提交记录，理解项目状态。先不要修改文件。读完后请总结当前工具已经完成什么、下一步适合做什么。
```

# 六、阶段 6：收尾、文档与最终提交

## 6-1. 完成全部设计文档内容后，应提交。新功能建议新开问答 session

当 `design.md` 中规划的内容全部完成后，应做一次阶段性提交。

提交前检查：

1. `design.md` 中的 Must have 功能是否完成。
2. 暂不实现内容是否没有被误做成半成品。
3. 项目能否从零安装和启动。
4. README 是否说明如何使用。
5. 是否有明显无用文件、临时代码、调试输出。

完成后提交 GitHub。之后如果要做新功能，建议新开 session，因为新功能往往会引入新的上下文和新的设计讨论。

## 6-2. 整体工具基本完成后，让 Codex 输出 QA_History.md 和 README.md

工具基本完成后，需要整理两个关键文档：

| 文档 | 作用 |
|---|---|
| QA_History.md | 总结开发过程中的关键问答、决策、问题和修复 |
| README.md | 给使用者看的项目说明、安装方式、使用方法和功能介绍 |

`QA_History.md` 不需要记录每一句对话，而是总结重要节点：

1. 项目最初目标。
2. 技术栈选择原因。
3. 核心功能如何确定。
4. 开发过程中遇到的重要问题。
5. 关键 bug 如何修复。
6. 哪些功能暂时不做。
7. 后续可以扩展什么。

提示词示例：

```text
请根据当前session的对话，总结输出 QA_History.md。需要逐字保留我的提问内容，codex的回复内容概括即可。
```

`README.md` 应包括：

1. 项目名称。
2. 项目简介。
3. 核心功能。
4. 技术栈。
5. 安装方法。
6. 启动方法。
7. 使用流程。
8. 示例输入输出。
9. 常见问题。
10. 后续计划。

提示词示例：

```text
请根据当前项目代码和 design.md 生成 README.md。要求面向第一次打开项目的人，说明这个工具是什么、如何安装、如何启动、如何使用、有哪些功能、当前限制是什么。
```

## 6-3. 最后做一轮提交 GitHub

最后一轮提交应包含（**实践课的最终作业内容应包括以下部分**）：

- 完整代码。
- `design.md`。
- `step-X.md`。
- `QA_History.md`。
- `README.md`。
- 必要的截图或演示材料。

最终提交前建议执行：

```text
请对当前项目做一次发布前检查：阅读 README.md、design.md 和代码，检查是否存在明显运行问题、文档不一致、临时文件、无用依赖或未说明的环境变量。先报告问题，再根据确认进行修复。
```

# 七、推荐文件结构

一个使用 Vibe Coding 开发的课程项目，建议保留以下文件：

```text
project-root/
  README.md
  design.md
  QA_History.md
  step-1.md
  step-2.md
  step-3.md
  step-4.md
  step-5.md
  （...其他具体程序代码...）
  .gitignore
```

不同项目会有不同目录，但建议始终保留：

| 文件 | 是否建议保留 | 原因 |
|---|---|---|
| design.md | 强烈建议 | 记录设计依据，防止上下文丢失 |
| step-X.md | 强烈建议 | 记录开发拆分，方便复盘和续做 |
| QA_History.md | 建议 | 记录关键决策和调试过程 |
| README.md | 必须 | 给别人和未来的自己使用 |
| .gitignore | 必须 | 避免提交依赖、缓存、密钥 |

# 八、常用提示词模板

## 构想阶段

Prompt：我想做一个工具，初步想法是：【写你的想法】。请帮我把它具体化，要求明确目标用户、输入、处理过程、输出、最小可用版本和暂不实现内容。先不要写代码。


## 调研总结阶段

Prompt：我调研了以下相似产品：【列出产品和观察】。请帮我总结它们的共性、可借鉴点、我这个课程项目第一版应该保留哪些功能、应该放弃哪些功能。

## 设计文档阶段

Prompt：

`/plan` 请基于以上讨论，为这个工具生成完整 design.md 草案。要求包含背景、用户、场景、产品形态、技术栈、功能清单、流程、数据结构、接口、错误处理、验收标准和暂不实现内容。


## 落盘 design.md

Prompt：请把刚才确认的设计方案写入项目根目录的 design.md。写入前请先说明文档结构，确认后再落盘。


## 拆分 step 文件

Prompt：请读取 design.md，把开发过程拆成 4-8 个 step-X.md。每个 step 必须包含目标、输入、输出、不做内容、实现要求、校验方式和完成标准。


## 执行某一步

Prompt：请执行 step-3.md。完成后按照 step-3.md 中的校验方式自动检查，并说明修改了哪些文件、校验是否通过、还有哪些风险。


## 调试问题


Prompt：

当前出现的问题是：【描述问题】。请先定位原因，不要立即大改。请阅读相关代码，说明可能原因、修改方案和风险，然后实施最小必要修复，并完成校验。


## UI 微调

```text
请微调当前页面：
1. 【问题 1】
2. 【问题 2】
3. 【问题 3】
4. 【问题 4】
完成后请检查桌面和移动端显示，说明改动文件和验证结果。
```

## 提交前审查

Prompt：请对当前工作区做提交前审查。重点检查功能是否跑通、是否有无关改动、是否有临时代码、README 是否与实际一致、是否有未处理的错误。先报告问题，不要直接提交。
 

## 提交前审查

Prompt：
请对当前工作区做提交前审查。重点检查功能是否跑通、是否有无关改动、是否有临时代码、README 是否与实际一致、是否有未处理的错误。先报告问题，不要直接提交。

## 提交/提交Github（注意下面的使用差别）

Prompt：
请提交当前代码。Commit为 “设计文档Design.md已经确定” 。

Prompt：
请提交当前代码到Github。Commit为 “设计文档Design.md已经确定” 。

**注意：** 如果不指出 `Github` ，那么codex会仅提交到本地Git，也可以作为备份，但只是本地备份，文件均存储在 `.git` 文件夹中。这里推荐直接提交到Github中，作为网络备份，一是更安全，二是查看代码也更为方便。

另外，Commit内容默认codex会生成，但作为解释注释当前这次提交内容的文本，推荐自己显式的写在提示词中。

## 总结开发历史

Prompt：
请根据当前对话、design.md、step-*.md 和项目代码，总结生成 QA_History.md。要求按阶段整理关键问答、设计决策、问题修复和经验，不要逐字记录聊天内容。

# 九、常见问题与建议

## 1. 为什么不能一开始就让 Codex 直接写完整项目？

因为工具开发不是单次问答。没有设计文档时，Codex 很容易按照自己的理解补全需求，结果可能和你真正想要的工具不一致。

正确做法是：

```text
想法具体化 -> 调研 -> /plan 设计 -> design.md -> step-X.md -> 分步实现 -> 调试 -> 文档 -> 提交
```

## 2. 为什么要写 design.md？

`design.md` 是项目的“共同记忆”。它能帮助你和 Codex 在长时间开发中保持方向一致。

尤其当出现以下情况时，`design.md` 很重要：

- 对话太长被压缩。
- 重新进入 Codex。
- 新开 session。
- 修改功能时需要判断是否偏离原设计。
- 提交前需要检查是否完成设计目标。

## 3. 为什么 step-X.md 要写输入和输出？

因为没有输入和输出，任务就不可验证。

差的步骤：

```text
优化上传功能。
```

好的步骤：

```text
读取 src/upload.ts 和 design.md，修复 PDF 上传后文本为空时没有错误提示的问题。输出应包括修改后的上传逻辑和页面错误提示。校验方式是上传一个空文件时页面显示明确错误，不崩溃。
```

## 4. 调试阶段为什么要 /resume？

调试依赖历史上下文。很多 bug 和设计选择有关，如果 Codex 不知道之前为什么这样写，就容易做出破坏性修改。

使用 `/resume` 可以延续原来的项目记忆，减少重复解释。

## 5. 什么时候应该新开 session？

适合新开 session 的条件是：

1. 当前阶段已经完成。
2. 已提交 GitHub。
3. README 或 QA_History.md 已更新。
4. 下一步是一个新功能或新阶段。

不适合在 bug 修到一半时新开 session。

## 6. 如何判断第一版是否完成？

第一版完成不等于完美。建议用以下标准判断：

| 标准 | 要求 |
|---|---|
| 能启动 | 安装依赖后项目可以运行 |
| 能输入 | 用户可以提供必要数据 |
| 能处理 | 核心逻辑能完成 |
| 能输出 | 用户能得到可用结果 |
| 能说明 | README 讲得清楚 |
| 能复现 | 别人按说明可以跑起来 |

## 7. 如何避免项目越做越大？

每次想加功能时，先问：

1. 这是不是 `design.md` 的 Must have？
2. 没有它，第一版还能不能用？
3. 这个功能是否会引入新技术栈？
4. 它是否值得新开一个阶段？

如果答案不清楚，就先写入“后续计划”，不要立刻实现。

# 总结

Vibe Coding 不是“随便让 AI 写代码”，而是一套持续协作流程：

1. 人负责提出真实问题、控制范围和判断质量。
2. Codex 负责辅助规划、生成代码、检查错误和整理文档。
3. 文档负责沉淀上下文，避免长对话失控。
4. GitHub 负责保存阶段成果，方便回退和复盘。

最关键的习惯是：

- 先 `/plan`，再 default。
- 先 `design.md`，再 `step-X.md`。
- 每步有输入、输出和校验。
- 调试阶段坚持 `/resume`。
- 阶段完成再提交。
- 最后整理 `QA_History.md` 和 `README.md`。

按照这个流程推进，即使项目一开始很小，也可以逐步 Vibe 成一个结构清楚、功能可用、文档完整的成熟工具。
